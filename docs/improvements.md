# Planned Improvements

## Critical (must fix before demo)

### 1. Context Window Management
- **Problem:** The agent loop appends messages indefinitely. Qwen3-32B has 32k context. A task reading multiple large files will exceed this and crash mid-task with no recovery.
- **Fix:** Three-layer defense:

  **Layer 1 — Smart File Reading (prevent large inputs)**
  - Modify `read_file` tool to accept optional `start_line` and `end_line` parameters
  - Agent requests only the section it needs instead of entire files
  - Default: if no range given, return first 200 lines + a note like `"truncated — 850 lines total, use start_line/end_line to read more"`

  **Layer 2 — Tool Result Truncation (safety net)**
  - Cap any single tool result at ~4,000 tokens before appending to messages
  - If truncated, append a note: `"[output truncated at 4k tokens, 12k total — re-run with narrower scope]"`
  - Applies to all tools, not just `read_file` (test output can be huge too)

  **Layer 3 — Sliding Window Summarization (last resort)**
  - Before each Groq API call, estimate total token count of `messages`
  - If exceeding 80% of model limit (~25k tokens):
    1. Take all messages except: system prompt, last 4 tool call/result pairs, last assistant message
    2. Summarize them into one message: `"[Summary of prior work: read X files, modified Y, tests failed twice on Z, current approach is...]"`
    3. Replace the old messages with this summary
  - The summary itself is generated by a cheap fast call (or a local heuristic if avoiding an extra API hit)

- **Where:** `agent/orchestrator.py` — add `trim_context(messages, max_tokens)` called before every Groq request. Layer 1 lives in `agent/tools.py` (read_file schema). Layer 2 lives in `agent/tools.py` (dispatch wrapper).

### 2. Progress Feedback to Telegram
- **Problem:** User sends a task and gets silence for minutes. No way to know if it's working or stuck.
- **Fix:** Send periodic Telegram updates at key moments:
  - "Reading repository files..."
  - "Writing changes to `src/utils.py`..."
  - "Running tests (attempt 2/3)..."
  - "Tests passed. Awaiting your approval."
- **Where:** `agent/orchestrator.py` — emit status callbacks that `bot/commands.py` forwards to Telegram.

## High Priority (strong portfolio impact)

### 3. Demo Mode
- **Problem:** Reviewers need 4 API keys to try the project. Most won't bother.
- **Fix:** Add a `--demo` flag to `main.py` that:
  - Uses a bundled sample repo (small Python project with a failing test)
  - Mocks Telegram interaction via CLI stdin/stdout
  - Runs a pre-defined task: "fix the failing test in test_math.py"
  - Shows the full agent loop in the terminal
- **Where:** New file `demo/runner.py`, modify `main.py` to accept `--demo`.

### 4. Structured Task Timeline
- **Problem:** No observability into what the agent did during a task.
- **Fix:** Log a structured timeline as JSON:
  ```json
  {
    "task_id": "abc123",
    "events": [
      {"t": "2024-01-15T10:00:01Z", "action": "read_file", "target": "src/utils.py", "duration_ms": 120},
      {"t": "2024-01-15T10:00:03Z", "action": "write_file", "target": "src/utils.py", "duration_ms": 85},
      {"t": "2024-01-15T10:00:05Z", "action": "run_tests", "result": "fail", "duration_ms": 3400},
      {"t": "2024-01-15T10:00:09Z", "action": "write_file", "target": "src/utils.py", "duration_ms": 90},
      {"t": "2024-01-15T10:00:12Z", "action": "run_tests", "result": "pass", "duration_ms": 3100}
    ]
  }
  ```
- **Where:** `agent/orchestrator.py` — append to timeline list after each tool dispatch. Save alongside checkpoint.

### 5. Retry/Backoff Strategy from Day One
- **Problem:** CLAUDE.md defers `tenacity` to Phase 6, but API failures will happen during development too.
- **Fix:** Add retry decorators early:
  - Groq calls: exponential backoff, max 3 retries, retry on 429/500/503
  - GitHub API: exponential backoff, max 3 retries, retry on 403 (rate limit)/500/502
  - E2B: exponential backoff, max 2 retries, retry on timeout/500
- **Where:** Wrap at the call site in each module, not in a shared utility.

## Medium Priority (nice to have)

### 6. Concurrent Task Awareness
- **Problem:** If two tasks are dispatched, they'll collide on sandbox state and branch names.
- **Fix:** Scope everything by `task_id` — branch names include task ID, each task gets its own sandbox instance, checkpoints are already per-task.
- **Where:** `agent/orchestrator.py`, `github/repo_manager.py`, `sandbox/e2b_runner.py`.

### 7. Simplify Memory Store
- **Problem:** ChromaDB is heavy for what will realistically be dozens of lessons per repo.
- **Fix:** Start with a simple JSON file per repo (`memory/{repo_slug}.json`) with keyword search. If it proves insufficient, migrate to ChromaDB later.
- **Where:** `memory/store.py` — implement JSON-backed store first, keep the interface so ChromaDB can be swapped in.

### 8. Better Architecture Diagram
- **Problem:** ASCII diagrams in CLAUDE.md don't render well everywhere.
- **Fix:** Add a Mermaid diagram in the README that GitHub renders natively:
  ```mermaid
  graph TD
    A[Telegram User] -->|task/voice| B[bot/handler]
    B -->|dispatch| C[agent/orchestrator]
    C -->|tool calls| D[github/repo_manager]
    C -->|run tests| E[sandbox/e2b_runner]
    C -->|recall lessons| F[memory/store]
    C -->|approval request| B
    B -->|approved| G[github/pr_manager]
    G -->|PR URL| B
    B -->|result| A
  ```
